When both `recital_date` and `credit_date` columns are present in the CSV headers, apply the calendar and date rules below. When either column is absent, keep milestone 2 matching and ignore the calendar file.

For this milestone, the verifier may add a `recital_date` column to `bookings.csv` and a `credit_date` column to `credits.csv`. A credit can match only when all prior criteria still pass, the credit_date is listed as `open` in `/app/config/cutoff_calendar.txt`, and the credit_date is not later than the booking recital_date. A missing or closed credit_date is not eligible. A booking with a missing recital_date is also not eligible because the date comparison cannot be satisfied. If more than one unused booking matches the same credit, choose the eligible booking with the latest recital_date; if recital_dates tie, choose the earliest booking row. Consumption is tracked by booking input row position, not by booking_id, so duplicate ids in separate rows remain independently consumable until their specific rows are used.

Legacy aliases from milestone 2 still apply (`SL` means `SOLO`, `GP` means `GROUP`, `ST` means `STAGE`), and matched report rows must emit the canonical recital_type. Unmatched rows must leave `recital_type` blank. Summary amounts remain positive integer cents.
