Extend the booking credit reconciler in `/app/cmd/reconcile/main.go` with configured type policy from `/app/config/methods.csv`, dancer daily credit limits from `/app/config/dancer_limits.csv`, and blackout-date exclusions from `/app/config/blackouts.csv` while preserving every milestone 1, milestone 2, and milestone 3 rule and output schema.

The methods file is a CSV with columns `recital_type`, `enabled`, and optional `priority`. Normalize `recital_type` after trimming and case folding using the same alias rules as credits (`SL` to `SOLO`, `GP` to `GROUP`, `ST` to `STAGE`). A selected canonical type is eligible only when `methods.csv` contains a row whose normalized type is canonical and whose `enabled` value is `true` case-insensitively. Lower `priority` numbers rank earlier; missing or malformed priorities rank after configured numeric priorities.

Credits may use `ANY` as the `recital_type`. `ANY` is never emitted in the report. An `ANY` credit can match any enabled booking recital type that satisfies the booking id, dancer id, amount, active status, date, and consumption rules. For `ANY`, select the unused eligible booking with the latest `recital_date` when date columns are present; if class dates tie, select the lower configured class-type priority; if priority also ties, select the earliest booking input row. When date columns are absent, rank only by priority then earliest booking row. Non-`ANY` credits still require the canonical recital type to match exactly and must be enabled in the methods file.

`dancer_limits.csv` is a CSV with columns `dancer_id,recital_type,effective_date,max_daily_amount,status`. Normalize `recital_type` after trimming and case folding with the same aliases as credits and methods. A dated action can match only if all prior gates pass and an `ACTIVE` limit row matches the action identity field and selected canonical type, has numeric `max_daily_amount`, and has `effective_date` on or before the action date. Rows that do not satisfy that definition do not apply. If multiple limit rows match, choose the one with the latest `effective_date`; if dates tie, choose the earliest limit input row.

The chosen limit is a daily cap for matched credits with the same `dancer_id`, selected canonical recital type, and `credit_date`. Matched credits consume that cap in credit input order. A credit that would make the consumed total exceed `max_daily_amount` must remain `UNMATCHED` and must not consume a booking row or budget. `ANY` credits use the recital type of the selected candidate booking for limit lookup and budget consumption. Apply the budget gate only in dated mode when `credit_date` exists; undated inputs continue to follow methods and `ANY` behavior without dancer-limit checks.

`blackouts.csv` is a CSV with columns `recital_type,start_date,end_date,state`. Normalize `recital_type` after trimming and case folding with the same aliases as credits, methods, and limits. A dated action cannot match a source when the selected canonical type has an `ACTIVE` blackout row with valid `YYYY-MM-DD` `start_date` and `end_date` and `start_date <= action date <= end_date`. Rows that do not satisfy that definition do not apply. If any valid active blackout applies, that candidate booking is ineligible.

For `ANY` credits, blackout filtering happens before candidate ranking by latest `recital_date`, configured priority, and earliest booking row. If the highest-priority or latest-date candidate is blacked out, choose the next eligible candidate that still satisfies every prior gate, method policy, calendar rule, and dancer limit. A credit rejected only because all otherwise valid candidates are blacked out remains `UNMATCHED` and consumes no booking row or budget.

When date columns are absent, skip blackout and dancer-limit gates and keep methods and `ANY` undated behavior. Continue to use `/app/config/cutoff_calendar.txt` as the open-date file, write `/app/out/recital_credit_report.csv` with columns `booking_id,dancer_id,recital_type,amount_cents,status`, use only `MATCHED` and `UNMATCHED`, leave `recital_type` blank on unmatched rows, and keep `/app/out/recital_credit_summary.json` with `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents` as positive integers.

Keep the deliverable as a Go CLI compiled with `/usr/local/go/bin/go`.
