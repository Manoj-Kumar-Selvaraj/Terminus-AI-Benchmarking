The sponsorship reconciliation CLI in `/app/cmd/reconcile/main.go` is misclassifying a few same-day rebates. GOLD rebates are being left unmatched, one sponsorship id collision is matching the wrong row, and the matched total is coming out with the wrong sign. Fix the Go program so it reads `/app/data/sponsorships.csv` and `/app/data/rebates.csv`, then writes `/app/out/rebate_report.csv` and `/app/out/rebate_summary.json`. A rebate matches only when sponsorship_id, sponsor_id, amount, sponsorship status, and allowed level all line up. In `sponsorships.csv`, the only eligible sponsorship status is the literal value `SIGNED` compared case-insensitively. Do not treat `POSTED`, `DRAFT`, or any other status string as eligible. Levels `BRONZE`, `GOLD`, and `PLATINUM` are allowed. Each sponsorship can be matched at most once; if duplicate rebates point at the same sponsorship, only the earliest eligible rebate consumes it. Input fields can have incidental surrounding spaces, and level and status comparisons should be case-insensitive. The report must keep one row per rebate in input order with columns `sponsorship_id,sponsor_id,level,amount_cents,status`. The `status` column in the report uses only `MATCHED` or `UNMATCHED` (never `SIGNED` or any sponsorship status). Use the rebate's level when matched and leave `level` blank when no sponsorship matched. Report fields should not carry incidental surrounding spaces from the input. The summary JSON must contain integer fields `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents`, with rebate amounts counted as positive cents.
