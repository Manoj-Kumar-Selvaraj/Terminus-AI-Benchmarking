Extend the sponsorship reconciliation CLI in `/app/cmd/reconcile/main.go` to handle dated rebate batches. It must still read `/app/data/sponsorships.csv` and `/app/data/rebates.csv`, then write `/app/out/rebate_report.csv` and `/app/out/rebate_summary.json`. Keep the report schema `sponsorship_id,sponsor_id,level,amount_cents,status` with `status` limited to `MATCHED` or `UNMATCHED` only, blank `level` on unmatched rows, and summary JSON fields `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents` as positive integer cents. **Backward compatibility:** Date rules apply only when **both** files include their date column (`event_end` in sponsorships and `rebate_date` in rebates). If either column is absent, ignore `/app/config/cutoff_calendar.txt` and match exactly as before (aliases, consumption, and all prior gates still apply). When both date columns are present, a rebate can match only when all prior criteria still pass, including sponsorship status `SIGNED` compared case-insensitively, `rebate_date` is listed as `open` in `/app/config/cutoff_calendar.txt`, and `rebate_date` is not later than the sponsorship `event_end`. Missing, closed, or unlisted `rebate_date` values are not eligible. A sponsorship with missing `event_end` is not eligible. If multiple unused sponsorship rows match one rebate, choose the row with the latest `event_end`. If `event_end` values tie, choose the earliest sponsorship input row (lowest row position in `sponsorships.csv`). Consumption is by row position, not `sponsorship_id`, so duplicate ids in separate rows remain separate. Later rebates in input order see only unconsumed rows; a wrong row choice can leave no eligible sponsorship for a later rebate. Legacy aliases still apply (`BRZ` means `BRONZE`, `GLD` means `GOLD`, `PLT` means `PLATINUM`), and matched report rows must emit the canonical level.
