When both `class_date` and `credit_date` columns are present in the CSV headers, apply the calendar and date rules below. When either column is absent, keep milestone 2 matching and ignore the calendar file.

For this milestone, the verifier may add a `class_date` column to `classes.csv` and a `credit_date` column to `credits.csv`. A credit can match only when all prior criteria still pass, the credit_date is listed as `open` in `/app/config/cutoff_calendar.txt`, and the credit_date is not later than the class class_date. A missing or closed credit_date is not eligible. A class with a missing class_date is also not eligible because the date comparison cannot be satisfied. If more than one unused class matches the same credit, choose the eligible class with the latest class_date; if class_dates tie, choose the earliest class row. Consumption is tracked by class input row position, not by class_id, so duplicate ids in separate rows remain independently consumable until their specific rows are used.

Legacy aliases from milestone 2 still apply (`FL` means `FLOW`, `PW` means `POWER`, `PR` means `PRIVATE`), and matched report rows must emit the canonical class_type. Unmatched rows must leave `class_type` blank. Summary amounts remain positive integer cents.
