Extend the tour refund reconciler in `/app/cmd/reconcile/main.go` with configured type policy from `/app/config/methods.csv`, guest daily refund limits from `/app/config/member_limits.csv`, and blackout-date exclusions from `/app/config/blackouts.csv` while preserving every milestone 1, milestone 2, and milestone 3 rule and output schema.

The methods file is a CSV with columns `tour_type`, `enabled`, and optional `priority`. Normalize `tour_type` after trimming and case folding using the same alias rules as refunds (`RF` to `REEF`, `SH` to `SHARK`, `VP` to `VIP`). A selected canonical type is eligible only when `methods.csv` contains a row whose normalized type is canonical and whose `enabled` value is `true` case-insensitively. Lower `priority` numbers rank earlier; missing or malformed priorities rank after configured numeric priorities.

Refunds may use `ANY` as the `tour_type`. `ANY` is never emitted in the report. An `ANY` refund can match any enabled tour type that satisfies the tour id, guest id, amount, active status, date, and consumption rules. For `ANY`, select the unused eligible tour with the latest `tour_date` when date columns are present; if class dates tie, select the lower configured class-type priority; if priority also ties, select the earliest tour input row. When date columns are absent, rank only by priority then earliest tour row. Non-`ANY` refunds still require the canonical tour type to match exactly and must be enabled in the methods file.

`member_limits.csv` is a CSV with columns `guest_id,tour_type,effective_date,max_daily_amount,status`. Normalize `tour_type` after trimming and case folding with the same aliases as refunds and methods. A dated action can match only if all prior gates pass and an `ACTIVE` limit row matches the action identity field and selected canonical type, has numeric `max_daily_amount`, and has `effective_date` on or before the action date. Rows that do not satisfy that definition do not apply. If multiple limit rows match, choose the one with the latest `effective_date`; if dates tie, choose the earliest limit input row.

The chosen limit is a daily cap for matched refunds with the same `guest_id`, selected canonical tour type, and `refund_date`. Matched refunds consume that cap in refund input order. A refund that would make the consumed total exceed `max_daily_amount` must remain `UNMATCHED` and must not consume a tour row or budget. `ANY` refunds use the tour type of the selected candidate tour for limit lookup and budget consumption. Apply the budget gate only in dated mode when `refund_date` exists; undated inputs continue to follow methods and `ANY` behavior without guest-limit checks.

`blackouts.csv` is a CSV with columns `tour_type,start_date,end_date,state`. Normalize `tour_type` after trimming and case folding with the same aliases as refunds, methods, and limits. A dated action cannot match a source when the selected canonical type has an `ACTIVE` blackout row with valid `YYYY-MM-DD` `start_date` and `end_date` and `start_date <= action date <= end_date`. Rows that do not satisfy that definition do not apply. If any valid active blackout applies, that candidate tour is ineligible.

For `ANY` refunds, blackout filtering happens before candidate ranking by latest `tour_date`, configured priority, and earliest tour row. If the highest-priority or latest-date candidate is blacked out, choose the next eligible candidate that still satisfies every prior gate, method policy, calendar rule, and guest limit. A refund rejected only because all otherwise valid candidates are blacked out remains `UNMATCHED` and consumes no tour row or budget.

When date columns are absent, skip blackout and guest-limit gates and keep methods and `ANY` undated behavior. Continue to use `/app/config/cutoff_calendar.txt` as the open-date file, write `/app/out/tour_refund_report.csv` with columns `tour_id,guest_id,tour_type,amount_cents,status`, use only `MATCHED` and `UNMATCHED`, leave `tour_type` blank on unmatched rows, and keep `/app/out/tour_refund_summary.json` with `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents` as positive integers.
