# Recover uncertain issuance across Vault failover

Single-node lifecycle handling is stable, but the failover exercise still creates duplicate or orphaned PostgreSQL identities. `/app/evidence/failover-issuance-trace.log` captures one request crossing a node epoch after the database user was created. Other reproductions lose the response before or after durable lease commit, restart with a torn journal tail, or retry a request whose outcome is unknown.

Repair credential issuance and restart reconciliation so the documented issuance operation remains one logical operation across retry, process restart, active-node change, and uncertain response. Identical replay must return or adopt the existing credential. Conflicting reuse of the operation identity must be rejected. Orphaned users and incomplete leases must be reconciled without privilege expansion or destructive global cleanup. Only the current active node and epoch may commit; stale-node outcomes cannot overwrite a recovered result (`STANDBY_NODE`, `STALE_EPOCH`). One stuck request must not block independent pods or request IDs. The verifier may drive issuance through trusted runtime commands (`issue`, `seed --fixture orphan-lease`) and failure injection points such as `USERNAME_COLLISION`. A torn final journal record may be recovered as documented, while malformed committed history must fail safely.

Preserve milestones 1–3, stable lease and request identities, role and privilege contracts, CLI schemas, pool behavior, and all unrelated state. Do not generate a new user on every retry, reset request history during failover, fabricate simulator files, hardcode generated identifiers, disable standby behavior, or solve concurrency with a process-wide lifecycle lock.
