# Recover renewable leases without exceeding their lifetime

Workload login is reliable, but long-running payment pods still lose database access around lease boundaries. `/app/evidence/lease-expiry-trace.log` and `/app/evidence/renewal-loop.log` show late attempts, unnecessary renewal traffic, replacement users created during transient errors, and expiry moving beyond the database role lifetime. Review `/app/docs/lease-lifecycle-contract.md` and reproduce the incident with the deterministic runtime clock rather than wall-clock delays.

Repair lease maintenance so the existing credential is renewed only during its permitted window, no more frequently than policy allows, and never beyond its maximum expiry. Renewal and its retries must preserve the lease ID, database username, owner, and generation; they must not issue another database user. Attempts inside the safety window but sooner than the configured minimum renewal interval must return `NO_ACTION` without creating another database user; the durable `last_renewed_at` field drives that throttling. Transient runtime failures require bounded deterministic retry while the still-valid lease remains usable, and a lease that cannot legally renew must produce the documented rotation-required outcome. Restarted processes must reconstruct usable lease state from durable application state and journal data, reject malformed durable timestamps with `MALFORMED_LEASE`, reject clock regression with `CLOCK_REGRESSION`, and never return expired or revoked credentials as active. Editable lifecycle code must not call `time.Sleep(` or `time.Now(` for scheduling.

Preserve milestone 1 behavior, the deterministic clock interface, CLI and JSON compatibility, lease policy values, stable IDs, unrelated lease independence, and existing state. Do not sleep until expiry, use uncontrolled wall-clock time, extend TTLs, delete state, disable expiry, or substitute static credentials.
