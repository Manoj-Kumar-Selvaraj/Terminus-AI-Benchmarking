# Complete mixed-version rollout and ownership-safe recovery

Failover replay is correct for one client generation, but the paused production rollout contains protocol version 1 and version 2 pods at the same time. Operator evidence shows one pod shutting down another pod's credential, stale generations attempting renewal, and restarts during pool creation, swap, and old-lease cleanup. Review `/app/docs/failover-recovery-contract.md`, `/app/docs/connection-rotation-contract.md`, and the complete incident timeline.

Finish the application so both documented response protocols coexist without exposing credentials. Version 1 must retain its exact legacy response fields; version 2 must receive the extended issuance, generation, and owner metadata. Every lease lifecycle action is scoped to the authenticated pod that owns it. Version 2 issuance and rotation require a non-empty request `pod_uid`; an empty owner is rejected with `MISSING_OWNER`. A version-2 request with an empty `request_id` is rejected with `INVALID_REQUEST`. A valid token cannot act on behalf of another pod UID (`WORKLOAD_OWNERSHIP_DENIED`). Renewing or revoking another pod's lease is rejected with `LEASE_OWNERSHIP_DENIED`. A stale pod or superseded generation cannot renew or revoke another active identity; renewing a superseded lease is rejected with `SUPERSEDED_LEASE` or `LEASE_REVOKED`. Restart during any rotation phase must select one valid active generation, preserve a healthy old pool until replacement is ready, and eventually finish cleanup. Legacy and extended lease records must restore together, rollout and rollback must remain available while a valid credential exists, and repeated completed reconciliation must make no further changes.

Password values and raw tokens must never appear in logs, journals, checkpoints, application snapshots, or the trusted runtime state directory. Preserve all earlier security, failover, least-privilege, deterministic-clock, CLI, and state contracts. When no valid dynamic credential exists, fail safely rather than using the static break-glass reference or weakening authorization.
