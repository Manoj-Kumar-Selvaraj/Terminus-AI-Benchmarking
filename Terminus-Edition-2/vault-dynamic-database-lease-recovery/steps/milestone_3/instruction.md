# Restore safe pool rotation and database revocation

Renewal now behaves correctly, but `/app/evidence/connection-pool-errors.log`, `/app/evidence/revocation-failure.log`, and `/app/evidence/database-user-leak-report.json` show revoked users authenticating, retired pools accepting new work, and one failed cleanup blocking other leases. Operators also observed a candidate pool replacing a healthy pool before its credential was usable.

Repair the real rotation, database-operation, shutdown, and cleanup workflows. A new dynamic credential must be usable before it can become the active pool generation. A failed candidate must leave the prior healthy generation serving traffic. After a successful swap, new work must use only the new generation; documented in-flight sessions may finish only within the drain policy. The old lease is revoked after the new pool is healthy, and its database user must reject new authentication. Partial rotation and restart must converge to one valid active generation. Failed revocations remain retryable and isolated so unrelated cleanup continues. The dynamic database role must enforce the allowed payment operations and deny role creation, schema changes, unrelated identity data, and cross-tenant access by executing the trusted simulated database operation (`run-db-operation`). Application source must not embed static-fallback literals such as `static_password`, `emergency_password`, `fallback-password`, `static-breakglass`, `payment_static`, `secret/static`, or `static_fallback`.

Preserve earlier authentication and renewal behavior, public interfaces, pool metadata, least privilege, and durable state. The break-glass reference is not an application fallback. Do not grant wildcard privileges, mark revocation successful without a runtime operation, delete all users, suppress fault injection, or serialize unrelated lifecycle work globally.
