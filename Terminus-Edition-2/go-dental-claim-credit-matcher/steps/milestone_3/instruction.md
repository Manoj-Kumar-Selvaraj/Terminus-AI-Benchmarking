Keep the deliverable as a Go CLI: the verifier compiles the Go source with the Go toolchain available at `/usr/local/go/bin/go` and then runs the produced binary.

For this milestone, the verifier supplies `claims.csv` with a trailing `service_date` column and `credits.csv` with a trailing `credit_date` column; those exact header names are the dated input schema used in tests. A credit can match only when all prior criteria still pass, including eligible claim status `APPROVED` compared case-insensitively, the credit_date is listed as `open` in `/app/config/cutoff_calendar.txt`, and the credit_date is not later than the claim service_date. Malformed `service_date` values use the same `YYYY-MM-DD` numeric-shape rules as credit dates and are not eligible. Calendar entries are split on whitespace; both the date token and state token are trimmed before comparison (for example, `  2026-04-05   open  ` is eligible). Verifier scenarios include tied `service_date` rows where two of three credits match and the third stays unmatched, and dated credits may use milestone 2 aliases such as `REST`. Calendar state is compared case-insensitively after trimming (for example, `OpEn` and `open` both mean open, and `  closed  ` means closed). Dates must use `YYYY-MM-DD` shape with numeric year, month `01` through `12`, and day `01` through `31`; non-numeric month or day characters (for example `2026-ab-05`) are malformed; malformed values such as `0000-00-00` are not eligible even if listed as open. A missing, closed, malformed, or unlisted credit_date is not eligible. A claim with a missing service_date is also not eligible because the date comparison cannot be satisfied. If more than one unused claim matches the same credit, choose the eligible claim with the latest service_date; if service_dates tie, choose the earliest claim row by input order in `claims.csv` (the first matching row in the file). Consumption is tracked by claim input row position, not by claim_id, so duplicate ids in separate rows remain independently consumable until their specific rows are used. When two credits share the same claim identity and two eligible rows differ only by `service_date`, the first credit should consume the latest `service_date` row and the second should consume the earlier row. Unmatched dated credits must trim `claim_id`, `patient_id`, and `amount_cents` in the report. The report must preserve credit input order.

Legacy aliases from milestone 2 still apply (`PREV` means `PREVENTIVE`, `REST` means `RESTORATIVE`, `ORT` means `ORTHO`), and matched report rows must emit the canonical procedure. Unmatched rows must leave `procedure` blank. Summary amounts remain positive integer cents. Milestone 1 and 2 rules still apply where relevant: full `claim_id` equality, trimming and case-insensitive `APPROVED` status and procedure handling, row-position consumption, credit input order, report field trimming on unmatched rows, the report header line exactly `claim_id,patient_id,procedure,amount_cents,status`, and summary JSON keys exactly `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents`.
