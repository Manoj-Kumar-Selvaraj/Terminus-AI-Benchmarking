Update the order reconciliation CLI in `/app` for a legacy adjustment export that uses service aliases. The program must still read `/app/data/orders.csv` and `/app/data/adjustments.csv`, then write `/app/out/adjustment_report.csv` and `/app/out/adjustment_summary.json`. An adjustment matches only when order_id, venue_id, amount, posted order status, and allowed service line up; order_ids must be compared as full identifiers, fields may have surrounding spaces, and service and status comparisons should be case-insensitive. Services `PICKUP`, `DELIVERY`, and `ONSITE` are allowed, and legacy adjustment service aliases `PU`, `DEL`, `OS` should be treated as canonical `PICKUP`, `DELIVERY`, and `ONSITE`.

Each order can be consumed by at most one adjustment, with the earliest eligible adjustment winning. The report must keep adjustment input order with columns `order_id,venue_id,service,amount_cents,status`; matched rows should emit the canonical service (`PICKUP`, `DELIVERY`, or `ONSITE`), while unmatched rows must leave `service` blank. The summary JSON must contain integer fields `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents`, with adjustment amounts counted as positive cents.
