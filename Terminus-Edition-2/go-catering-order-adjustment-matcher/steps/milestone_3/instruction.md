Extend the order reconciliation CLI in `/app/cmd/reconcile/main.go` to handle dated adjustment batches. It must still read `/app/data/orders.csv` and `/app/data/adjustments.csv`, then write `/app/out/adjustment_report.csv` and `/app/out/adjustment_summary.json` with the same schema and status values from the earlier milestones.

The input files may add a `fulfill_date` column to `orders.csv` and an `adjustment_date` column to `adjustments.csv`. An adjustment can match only when all prior criteria still pass (order_id, venue_id, amount, `FULFILLED` status, and allowed service alignment), the adjustment_date is listed as `open` in `/app/config/cutoff_calendar.txt`, and the adjustment_date is not later than the order fulfill_date (i.e., adjustment_date <= fulfill_date). A missing or closed adjustment_date is not eligible. An order with a missing fulfill_date is also not eligible because the date comparison cannot be satisfied. If more than one unused order matches the same adjustment, choose the eligible order with the latest fulfill_date; if fulfill_dates tie, choose the earliest order row. Consumption is tracked by order input row position, not by order_id, so duplicate ids in separate rows remain independently consumable until their specific rows are used.

Legacy aliases from milestone 2 still apply (`PU` means `PICKUP`, `DEL` means `DELIVERY`, `OS` means `ONSITE`), and matched report rows must emit the canonical service. Unmatched rows must leave `service` blank. Summary amounts remain positive integer cents.
