The order reconciliation CLI in `/app/cmd/reconcile/main.go` is misclassifying a few same-day adjustments. DELIVERY adjustments are being left unmatched, one order id collision is matching the wrong row, and the matched total is coming out with the wrong sign.

Fix the Go program so it reads `/app/data/orders.csv` and `/app/data/adjustments.csv`, then writes `/app/out/adjustment_report.csv` and `/app/out/adjustment_summary.json`. An adjustment matches only when order_id, venue_id, amount, `FULFILLED` order status (compared case-insensitively after trimming; values such as `DRAFT` are not eligible), and an allowed service all line up. Services `PICKUP`, `DELIVERY`, and `ONSITE` are allowed. Each order can be matched at most once; if duplicate adjustments point at the same order, only the earliest eligible adjustment consumes it. Input fields can have incidental surrounding spaces, and service and status comparisons should be case-insensitive.

The report must keep one row per adjustment in input order with columns `order_id,venue_id,service,amount_cents,status`; use `MATCHED` or `UNMATCHED` for status, use the adjustment's service when matched, and leave `service` blank when no order matched. Report fields should not carry incidental surrounding spaces from the input. The summary JSON must contain integer fields `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents`, with adjustment amounts counted as positive cents.
