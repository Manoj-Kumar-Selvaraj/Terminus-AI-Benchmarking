Extend the stall reconciliation CLI in `/app/cmd/reconcile/main.go`. Keep every milestone 1–3 rule, including full-key matching, alias normalization, row-position consumption, cutoff-calendar gating, latest `market_date` selection with earliest-row tie breaks, blank `stall_type` on unmatched rows, positive summary totals, and the existing `/app/out/refund_report.csv` and `/app/out/refund_summary.json` schemas. Refunds input now includes a leading `settlement_id` column (`settlement_id,stall_id,vendor_id,amount_cents,stall_type` plus optional `refund_date`); parse by header name and ignore `settlement_id` for matching.

Milestone 4 adds `/app/config/stall_policy.csv` with columns `stall_type,enabled,priority`. Only canonical stall types whose policy row has `enabled` equal to `Y` are match-eligible. Policy names are canonical after trim and case folding. A refund with a concrete `stall_type` such as `PRODUCE`, `CRAFT`, or `FOOD`, or an alias that normalizes to one of those values, must still match the source canonical stall type exactly and must also pass the enabled-policy gate.

A refund with `stall_type` equal to `ANY` is a policy-driven refund: it may match any otherwise eligible unused stall row whose canonical `stall_type` is enabled. If more than one stall row qualifies for an `ANY` refund, choose the row with the latest `market_date` when date columns are in use; if `market_date` values tie or both sides are blank on the dated columns, choose the lowest configured `priority` number; if priority also ties, choose the earliest stall input row. Matched `ANY` rows report the selected source row's canonical `stall_type`; never emit `ANY` in the report.
