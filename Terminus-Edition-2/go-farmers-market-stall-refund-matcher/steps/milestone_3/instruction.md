Extend the stall reconciliation CLI in `/app/cmd/reconcile/main.go` to handle dated refund batches. It must still read `/app/data/stalls.csv` and `/app/data/refunds.csv`, then write `/app/out/refund_report.csv` and `/app/out/refund_summary.json` with the same schema and status values from the earlier milestones. input files may include a `market_date` column in `stalls.csv` and a `refund_date` column in `refunds.csv`. A refund can match only when all prior criteria still pass, including the requirement that the stall row status is `RESERVED` (case-insensitive), the refund_date is listed as `open` in `/app/config/cutoff_calendar.txt`, and the refund_date is not later than the stall market_date. When both `market_date` and `refund_date` are blank on a candidate pair, treat the row like the undated milestone-2 path and skip calendar and date-order gates. A missing or closed refund_date is not eligible when a date comparison is required. A stall with a missing market_date is also not eligible when a date comparison is required. If more than one unused stall matches the same refund, choose the eligible stall with the latest market_date; if market_dates tie, choose the earliest stall row. Consumption is tracked by stall input row position, not by stall_id, so duplicate ids in separate rows remain independently consumable until their specific rows are used. Legacy aliases from milestone 2 still apply (`PRD` means `PRODUCE`, `CRT` means `CRAFT`, `FOD` means `FOOD`), and matched report rows must emit the canonical stall_type. Unmatched rows must leave `stall_type` blank. Summary amounts remain positive integer cents.
