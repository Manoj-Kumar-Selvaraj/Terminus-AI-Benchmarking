Extend the wire-return batch in `/app/src/wire_returns.cbl` with reason-code policy from `/app/config/reason_codes.csv`. Keep every prior requirement: full wire-id match, one wire can be consumed once, settled status, settlement-cycle window, latest-date wire selection, and tie-break by earliest wire row.

When `/app/config/reason_codes.csv` is present it includes `code`, `meaning`, `enabled`, and `priority` columns. A disabled reason code must not clear even if it was allowed by earlier milestones. The `priority` column contains a positive integer; lower numbers rank earlier. Missing or malformed priorities rank after all configured numeric priorities.

Return records may carry a 3-character reason code at positions 40-42 (immediately after the optional 8-character return date at positions 32-39). When that field is `ANY`, the return can clear against any eligible wire whose reason code is enabled in the policy. For `ANY`, if more than one eligible wire remains after all prior gates, select the one with the latest settlement date; when dates tie, select the wire with the lower configured reason-code priority; when priority also ties, select the earliest wire input row. Non-`ANY` returns still require the wire reason to match the return reason exactly and the reason to be enabled. A blank reason field at positions 40-42 behaves as before milestone 4: it matches any enabled reason code without priority ordering.

The report must emit the matched wire's canonical reason code for every `CLEARED` row; never emit `ANY` in the report. `EXCEPTION` rows must leave the reason field blank. Keep all output schemas and summary keys from earlier milestones.
