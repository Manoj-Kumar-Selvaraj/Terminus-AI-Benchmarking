Finish the wire-return batch in `/app/src/wire_returns.cbl` for settlement-cycle controls. Wire records may include an 8-character settlement date immediately after the settled-status field, and return records may include an 8-character return date immediately after the account field. Use `/app/config/cycle_calendar.txt`, where only rows marked `OPEN` are settlement-cycle days. Calendar status tokens are case-insensitive (`open`, `Open`, and `OPEN` all mean open), and calendar rows may appear in any order.

A return can clear only when the wire settlement date and return date are nonblank, present in the calendar as `OPEN`, the return date is not before the wire settlement date, and the return arrives within two open cycle days after the wire settlement date. Count open days after the wire settlement date through the return date, including the return date if it is open and excluding closed or absent calendar dates. Exactly two open cycle days is still eligible; three or more is not. Dates missing from the calendar are not open. When only one side has a blank settlement or return date, treat the pair as ineligible for clearing. When both the wire settlement date and return date are blank on a candidate pair, treat the row like the undated milestone-2 path and skip calendar and cycle-window gates. If more than one unused wire satisfies all matching and cycle-window rules for a return, clear against the eligible wire with the latest settlement date; when dates tie, use the earliest wire input row. Keep all prior behavior and output schemas for `/app/out/wire_return_report.csv` and `/app/out/wire_return_summary.txt`; `EXCEPTION` rows must leave the `reason` field blank.
