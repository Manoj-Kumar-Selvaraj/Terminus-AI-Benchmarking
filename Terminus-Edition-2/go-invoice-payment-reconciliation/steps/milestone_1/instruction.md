The reconciliation CLI under `/app` must read `/app/data/invoices.csv` and `/app/data/payments.csv`, then write `/app/out/payment_report.csv` and `/app/out/payment_summary.json`. Keep the deliverable as a Go CLI: the verifier compiles the Go source with `/usr/local/go/bin/go` and runs the produced binary.

For milestone 1, input headers are `invoice_id,customer_id,amount_cents,status,method` for invoices and `invoice_id,customer_id,amount_cents,method` for payments. A payment matches only when the cleaned `invoice_id`, cleaned `customer_id`, positive integer `amount_cents`, `POSTED` invoice status, and canonical method all match. Allowed methods are `ACH`, `CARD`, and `WIRE`. Method and status comparisons are case-insensitive, and incidental surrounding whitespace on input fields must be ignored. Non-positive, signed, decimal, blank, or non-numeric amount values make that row ineligible; invalid payment amounts still produce an `UNMATCHED` report row and contribute `0` to `unmatched_amount_cents`.

Each invoice input row can be consumed at most once. If more than one invoice row satisfies the milestone 1 contract for a payment, use the first qualifying invoice row in file order. The report must keep one row per payment in input order with columns `invoice_id,customer_id,method,amount_cents,status`; use `MATCHED` or `UNMATCHED`, write the canonical method on matched rows, and leave `method` blank on unmatched rows. The summary JSON must contain integer fields `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents`, with payment amounts counted as positive cents.
