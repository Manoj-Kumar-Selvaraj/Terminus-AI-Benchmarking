Fix the courier COD remittance reconciler in `/app/app/reconcile.rb`. The deliverable is a Ruby CLI; run it with `ruby /app/app/reconcile.rb`. Moving the implementation to Python or another uninvoked file does not satisfy the task. It must read `/app/data/deliveries.csv`, `/app/data/remittances.csv`, and `/app/config/windows.csv`, then write `/app/out/cod_remittance_report.csv` and `/app/out/cod_remittance_summary.txt`. See `/app/docs/record_layouts.md` for column names and the baseline matching contract.

A correction matches a delivery only when `parcel_id`, `courier_id`, `station_id`, `location`, and `amount` all match exactly, the delivery status is `DELIVERED`, the correction reason is `RETURN`, `SHORT`, or `ADJUST`, the `kind` is `CASH`, `UPI`, or `CARD`, both timestamps are 14-digit numeric values, and `action_ts` is on or after `source_ts`. The delivery timestamp must fall inside an `OPEN` window for the same `station_id`, and the correction timestamp must not exceed that window close. Each delivery row can be consumed once. When multiple unused deliveries qualify, choose the latest `source_ts`; if timestamps tie, choose the earliest delivery input row.

Write the report with columns `action_id,parcel_id,courier_id,station_id,kind,amount,reason,status`, preserving correction input order. Matched rows emit canonical `kind`; unmatched rows leave `kind` blank. Use only `MATCHED` or `UNMATCHED`. Write the summary as `matched_count`, `matched_amount`, `unmatched_count`, and `unmatched_amount` key=value lines with non-negative integer amounts (zero when no rows qualify in that bucket).
