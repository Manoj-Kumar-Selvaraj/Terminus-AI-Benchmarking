Extend the pass reconciliation CLI in `/app/cmd/reconcile/main.go` to handle dated refund batches. It must still read `/app/data/passes.csv` and `/app/data/refunds.csv`, then write `/app/out/refund_report.csv` and `/app/out/refund_summary.json` with the same schema and status values from the earlier milestones. When both `visit_date` and `refund_date` columns are present in the CSV headers, apply the calendar and date rules below. When either column is absent, keep milestone 2 matching and ignore the calendar file.

For this milestone, the verifier may add a `visit_date` column to `passes.csv` and a `refund_date` column to `refunds.csv`. A refund can match only when all prior criteria still pass, the refund_date is listed as `open` in `/app/config/cutoff_calendar.txt`, and the refund_date is not later than the pass visit_date. A missing or closed refund_date is not eligible. A pass with a missing visit_date is also not eligible because the date comparison cannot be satisfied. If more than one unused pass matches the same refund, choose the eligible pass with the latest visit_date; if visit_dates tie, choose the earliest pass row. Consumption is tracked by pass input row position, not by pass_id, so duplicate ids in separate rows remain independently consumable until their specific rows are used.

Legacy aliases from milestone 2 still apply (`DY` means `DAY`, `SEA` means `SEASON`, `V` means `VIP`), and matched report rows must emit the canonical access_type. Unmatched rows must leave `access_type` blank. Summary amounts remain positive integer cents.
