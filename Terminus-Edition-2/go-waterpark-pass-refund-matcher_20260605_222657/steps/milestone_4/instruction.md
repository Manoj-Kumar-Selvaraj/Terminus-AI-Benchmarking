Extend the pass refund reconciler in `/app/cmd/reconcile/main.go` with access-type policy from `/app/config/methods.csv`, guest daily refund limits from `/app/config/guest_limits.csv`, and blackout-date exclusions from `/app/config/blackouts.csv` while preserving every milestone 1, milestone 2, and milestone 3 rule and output schema.

The methods file is a CSV with columns `access_type`, `enabled`, and optional `priority`. Normalize `access_type` after trimming and case folding using the same alias rules as refunds (`DY` to `DAY`, `SEA` to `SEASON`, `V` to `VIP`). A pass or refund access type is eligible only when it appears in `methods.csv` with `enabled` set to `true` case-insensitively. Disabled types, missing types, blank names, malformed rows, and non-`true` enabled values must not become match-eligible. Ignore malformed method rows instead of crashing. Lower `priority` numbers rank earlier; missing or malformed priorities rank after configured numeric priorities.

Refunds may use `ANY` as the `access_type`. `ANY` is never emitted in the report. An `ANY` refund can match any enabled pass access type that satisfies the pass id, guest id, amount, active status, date, and consumption rules. For `ANY`, select the unused eligible pass with the latest `visit_date` when date columns are present; if visit dates tie, select the lower configured access-type priority; if priority also ties, select the earliest pass input row. When date columns are absent, rank only by priority then earliest pass row. Non-`ANY` refunds still require the canonical access type to match exactly and must be enabled in the methods file.

`guest_limits.csv` is a CSV with columns `guest_id,access_type,effective_date,max_daily_amount,status`. Normalize `access_type` after trimming and case folding with the same aliases as refunds and methods. A dated refund can match only if all prior gates pass and an `ACTIVE` limit row matches the refund's `guest_id` and the selected canonical pass access type, has numeric `max_daily_amount`, and has `effective_date` on or before the refund's `refund_date`. Missing, malformed, inactive, unknown-access-type, future-effective, or nonnumeric limit rows are ignored. If multiple limit rows match, choose the one with the latest `effective_date`; if dates tie, choose the earliest limit input row.

The chosen limit is a daily cap for matched refunds with the same `guest_id`, selected canonical access type, and `refund_date`. Matched refunds consume that cap in refund input order. A refund that would make the consumed total exceed `max_daily_amount` must remain `UNMATCHED` and must not consume a pass row or budget. `ANY` refunds use the access type of the selected candidate pass for limit lookup and budget consumption. Apply the budget gate only in dated mode when `refund_date` exists; undated inputs continue to follow methods and `ANY` behavior without guest-limit checks.

`blackouts.csv` is a CSV with columns `access_type,start_date,end_date,state`. Normalize `access_type` after trimming and case folding with the same aliases as refunds, methods, and limits. A dated refund cannot match a pass when the selected canonical pass access type has an `ACTIVE` blackout whose `start_date` and `end_date` are valid `YYYY-MM-DD` dates and where `start_date <= refund_date <= end_date`. Inactive rows, unknown access types, blank access types, malformed dates, and rows whose range does not contain the refund date are ignored. If any valid active blackout applies, that candidate pass is ineligible.

For `ANY` refunds, blackout filtering happens before candidate ranking by latest `visit_date`, configured priority, and earliest pass row. If the highest-priority or latest-date candidate is blacked out, choose the next eligible candidate that still satisfies every prior gate, method policy, calendar rule, and guest limit. A refund rejected only because all otherwise valid candidates are blacked out remains `UNMATCHED` and consumes no pass row or budget.

When date columns are absent, skip blackout and guest-limit gates and keep methods and `ANY` undated behavior.
