Extend the ticket credit reconciler in `/app/cmd/reconcile/main.go` with configured type policy from `/app/config/methods.csv`, rider daily credit limits from `/app/config/rider_limits.csv`, and blackout-date exclusions from `/app/config/blackouts.csv` while preserving every milestone 1, milestone 2, and milestone 3 rule and output schema.

The methods file is a CSV with columns `fare_type`, `enabled`, and optional `priority`. Normalize `fare_type` after trimming and case folding using the same alias rules as credits (`EC` to `ECON`, `BK` to `BIKE`, `CB` to `CABIN`). A selected canonical type is eligible only when `methods.csv` contains a row whose normalized type is canonical and whose `enabled` value is `true` case-insensitively. Lower `priority` numbers rank earlier; missing or malformed priorities rank after configured numeric priorities.

Credits may use `ANY` as the `fare_type`. `ANY` is never emitted in the report. An `ANY` credit can match any enabled ticket fare type that satisfies the ticket id, rider id, amount, active status, date, and consumption rules. For `ANY`, select the unused eligible ticket with the latest `travel_date` when date columns are present; if class dates tie, select the lower configured class-type priority; if priority also ties, select the earliest ticket input row. When date columns are absent, rank only by priority then earliest ticket row. Non-`ANY` credits still require the canonical fare type to match exactly and must be enabled in the methods file.

`rider_limits.csv` is a CSV with columns `rider_id,fare_type,effective_date,max_daily_amount,status`. Normalize `fare_type` after trimming and case folding with the same aliases as credits and methods. A dated action can match only if all prior gates pass and an `ACTIVE` limit row matches the action identity field and selected canonical type, has numeric `max_daily_amount`, and has `effective_date` on or before the action date. Rows that do not satisfy that definition do not apply. If multiple limit rows match, choose the one with the latest `effective_date`; if dates tie, choose the earliest limit input row.

The chosen limit is a daily cap for matched credits with the same `rider_id`, selected canonical fare type, and `credit_date`. Matched credits consume that cap in credit input order. A credit that would make the consumed total exceed `max_daily_amount` must remain `UNMATCHED` and must not consume a ticket row or budget. `ANY` credits use the fare type of the selected candidate ticket for limit lookup and budget consumption. Apply the budget gate only in dated mode when `credit_date` exists; undated inputs continue to follow methods and `ANY` behavior without rider-limit checks.

`blackouts.csv` is a CSV with columns `fare_type,start_date,end_date,state`. Normalize `fare_type` after trimming and case folding with the same aliases as credits, methods, and limits. A dated action cannot match a source when the selected canonical type has an `ACTIVE` blackout row with valid `YYYY-MM-DD` `start_date` and `end_date` and `start_date <= action date <= end_date`. Rows that do not satisfy that definition do not apply. If any valid active blackout applies, that candidate ticket is ineligible.

For `ANY` credits, blackout filtering happens before candidate ranking by latest `travel_date`, configured priority, and earliest ticket row. If the highest-priority or latest-date candidate is blacked out, choose the next eligible candidate that still satisfies every prior gate, method policy, calendar rule, and rider limit. A credit rejected only because all otherwise valid candidates are blacked out remains `UNMATCHED` and consumes no ticket row or budget.

When date columns are absent, skip blackout and rider-limit gates and keep methods and `ANY` undated behavior. Continue to use `/app/config/cutoff_calendar.txt` as the open-date file, write `/app/out/ticket_credit_report.csv` with columns `ticket_id,rider_id,fare_type,amount_cents,status`, use only `MATCHED` and `UNMATCHED`, leave `fare_type` blank on unmatched rows, and keep `/app/out/ticket_credit_summary.json` with `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents` as positive integers.
