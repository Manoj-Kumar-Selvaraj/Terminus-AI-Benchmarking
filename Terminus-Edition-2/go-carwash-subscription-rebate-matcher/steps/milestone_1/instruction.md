The wash reconciliation CLI in `/app/cmd/reconcile/main.go` is misclassifying same-day rebates. PLUS credits are being left unmatched, one wash id collision is matching the wrong row, duplicate credits can reuse the same wash, the matched total has the wrong sign, and the summary is written to the wrong output path. Fix the Go program so it reads `/app/data/washes.csv` and `/app/data/rebates.csv`, then writes `/app/out/wash_rebate_report.csv` and `/app/out/wash_rebate_summary.json`. A rebate matches only when trimmed `wash_id`, trimmed `customer_id`, positive integer `amount_cents`, posted trip status, and an allowed `plan_tier` all line up. Posted trip status is `COMPLETED` in `washes.csv`. Canonical plan tiers `BASIC`, `PLUS`, and `PRO` are allowed. Each physical wash row can be matched at most once; if duplicate rebates point at the same wash, only the earliest eligible rebate consumes it. Input fields may have incidental surrounding spaces, and plan_tier and status comparisons are case-insensitive.

The report must keep one row per rebate in input order with columns `wash_id,customer_id,plan_tier,amount_cents,status`; use `MATCHED` or `UNMATCHED` for status, and leave `plan_tier` blank when no wash matched. Matched rows must emit the canonical rebate plan_tier from `rebates.csv`, not the wash record's stored plan_tier field. Report fields must not carry incidental surrounding spaces from the input. The summary JSON must contain integer fields `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents`, with rebate amounts counted as positive cents.
