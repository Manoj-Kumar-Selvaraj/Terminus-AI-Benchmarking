Extend the wash reconciliation CLI in `/app/cmd/reconcile/main.go` to handle dated rebate batches. It must still read `/app/data/washes.csv` and `/app/data/rebates.csv`, then write `/app/out/wash_rebate_report.csv` and `/app/out/wash_rebate_summary.json` with the same schema, status values, canonical plan_tier output, blank unmatched plan_tier output, and positive summary cents from the earlier milestones. Keep the deliverable as a Go CLI.

Input files may include a `wash_date` column in `washes.csv` and a `rebate_date` column in `rebates.csv`. If neither input file contains date columns, preserve the milestone 1 and milestone 2 undated behavior. If either date column is present, dated matching applies: a rebate can match only when all prior criteria still pass, the rebate_date is listed as `open` in `/app/config/cutoff_calendar.txt`, the rebate_date is a valid `YYYY-MM-DD` date, and the rebate_date is not later than the trip wash_date. A missing, blank, malformed, unlisted, or closed rebate_date is not eligible. A trip with a missing, blank, or malformed wash_date is also not eligible because the date comparison cannot be satisfied.

If more than one unused trip matches the same rebate, choose the eligible trip with the latest wash_date; if wash_dates tie, choose the earliest wash row. Consumption is tracked by physical wash input row position, not only by wash_id, so duplicate ids in separate rows remain independently consumable until their specific rows are used. Legacy aliases from milestone 2 still apply (`BS` means `BASIC`, `PL` means `PLUS`, `PR` means `PRO`, compared case-insensitively).
