Extend the wash reconciliation CLI in `/app/cmd/reconcile/main.go` to enforce customer-specific rebate limits. Preserve all milestone 1 through milestone 4 behavior, including exact identifiers, completed status, alias normalization, dated-batch rules, methods eligibility, latest wash_date selection, earliest-row tie-breaks, row-position consumption, report schema, canonical matched plan_tier output, blank unmatched plan_tier output, and positive summary cents.

The program must read `/app/config/customer_limits.csv` at runtime using schema `customer_id,plan_tier,max_rebate_cents,enabled`. Trim `customer_id`, trim and canonicalize `plan_tier` through the same canonical and alias rules as rebate inputs, parse `max_rebate_cents` as a non-negative integer, and parse `enabled=true` case-insensitively. A candidate wash is eligible only when there is a customer-limit row for the trimmed customer_id and canonical plan_tier, the row is enabled, and the rebate amount is less than or equal to `max_rebate_cents`.

For repeated rows for the same trimmed customer_id and canonical plan_tier, the last row in file order that names a supported canonical tier is authoritative, including a later disabled or invalid-limit row making that customer/tier ineligible. Missing customer-limit rows, blank customer ids, blank or unsupported tiers, malformed rows, missing enabled values, disabled or non-true enabled values, blank maximums, non-integer maximums, and negative maximums are ineligible. The customer-limit gate applies in both undated and dated modes after methods eligibility, and must not bypass any earlier matching, date, methods, row-consumption, report, or summary requirement.
