The order reconciliation CLI in `/app/cmd/reconcile/main.go` is misclassifying a few same-day credits. The verifier recompiles that source file with `/usr/local/go/bin/go`, runs the resulting binary, and does not grade edits made only to `/app/build/reconcile`. REGIONAL credits are being left unmatched, one order id collision is matching the wrong row, and the matched total is coming out with the wrong sign.

Fix the Go program so it reads `/app/data/orders.csv` and `/app/data/credits.csv`, then writes `/app/out/credit_report.csv` and `/app/out/credit_summary.json`. A credit matches only when order_id, cafe_id, amount, posted order status, and route all line up; the credit route must equal the order route, and both must be one of the allowed routes; posted order status is stored as `FULFILLED` in `orders.csv`. Routes `LOCAL`, `REGIONAL`, and `EXPORT` are allowed. A credit whose route is outside those allowed values is not match-eligible even if every other field lines up. Each order can be matched at most once; if duplicate credits point at the same order, only the earliest eligible credit consumes it. Input fields can have incidental surrounding spaces, and route and status comparisons should be case-insensitive.

The report must keep one row per credit in input order with columns `order_id,cafe_id,route,amount_cents,status`; use `MATCHED` or `UNMATCHED` for status, use the credit's route when matched, and leave `route` blank when no order matched. Report fields should not carry incidental surrounding spaces from the input. The summary JSON must contain integer fields `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents`, with credit amounts counted as positive cents.
