Extend the order reconciliation CLI in `/app/cmd/reconcile/main.go` to handle dated credit batches. It must still read `/app/data/orders.csv` and `/app/data/credits.csv`, then write `/app/out/credit_report.csv` and `/app/out/credit_summary.json` with the same schema and status values from the earlier milestones. Keep the deliverable as a Go CLI: the verifier compiles the Go source with the Go toolchain available at `/usr/local/go/bin/go` and then runs the produced binary.

For this milestone, input files may include a `bake_date` column in `orders.csv` and a `credit_date` column in `credits.csv`. If both files still use the earlier schemas without those columns, keep the prior matching behavior. A credit can match only when all prior criteria still pass (full `order_id`, `cafe_id`, `amount_cents`, source status exactly `FULFILLED`, and route equality after alias normalization), the `credit_date` is listed as `open` in `/app/config/cutoff_calendar.txt`, and the `credit_date` is not later than the order `bake_date`. The status comparison remains trim- and case-insensitive from earlier milestones, so an input value such as `fulfilled` is treated as the required `FULFILLED` status. Non-fulfilled statuses such as `PENDING` remain ineligible. When choosing among eligible orders, a later `bake_date` on a row that fails any prior criterion, including wrong `order_id`, wrong `cafe_id`, wrong amount, wrong status, or wrong route, must not win over an earlier row that satisfies every gate. A missing or closed `credit_date` is not eligible. An order with a missing `bake_date` is also not eligible because the date comparison cannot be satisfied.

Treat calendar `open` state case-insensitively after trimming whitespace. Skip malformed calendar lines silently rather than treating them as open dates. Each valid calendar row is exactly two whitespace-separated tokens: a date token followed by a state token. Rows with fewer than two tokens or more than two tokens are ignored.

If more than one unused order matches the same credit, choose the eligible order with the latest `bake_date`; if `bake_date` values tie, choose the earliest order input row. Consumption is tracked by order input row position, not by `order_id`, so duplicate ids in separate rows remain independently consumable until their specific rows are used.

Legacy aliases from milestone 2 still apply (`LOC` means `LOCAL`, `REG` means `REGIONAL`, `EXP` means `EXPORT`), and matched report rows must emit the canonical route. Unmatched rows must leave `route` blank. Summary amounts remain positive integer cents.
