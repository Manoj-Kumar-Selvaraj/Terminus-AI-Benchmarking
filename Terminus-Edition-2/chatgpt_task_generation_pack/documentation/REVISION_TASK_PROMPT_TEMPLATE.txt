STEP 1b — Test Quality review (mandatory)

  Review every steps/milestone_N/tests/test_mN.py and fix ALL test-quality issues from STB
  and agent_review.txt, including warnings and suggestions (not only hard fails).

  Check each test:
    [ ] Docstring matches what assertions actually verify (informative_test_structure)
    [ ] Tie-break / selection rules use discriminating fixtures (e.g. distinct amounts for
        latest-date wins — same-amount tests can pass with wrong first-fit logic)
    [ ] Consumption tests prove which row was consumed, not only MATCHED/UNMATCHED counts
    [ ] Milestone-scoped tests do not falsely require later-milestone behavior (add helper
        comments when shared fixtures write files ignored in early milestones)
    [ ] Every instruction requirement for that milestone has at least one focused test
    [ ] No test references verifier/harness wording in user-facing docstrings

  Add or tighten tests where agent_review or behavior_in_tests flags a gap. Re-run oracle
  after any test change.

STEP 1c — Rubric review (mandatory; paste into Snorkel UI only — never inside zip)

  Files (same content in both):
    - revision-custom-rubrics/{task-folder-name}.rubric.txt   ← user paste copy
    - {task-folder-name}/rubric.txt                           ← task authoring copy

  Reference: documentation/RUBRICS_GUIDE.md, documentation/GPT_AGENT_PLAYBOOK.md §6

  Milestone segregation (required in local files):
    [ ] One clearly separated block per milestone, headed exactly `# Rubric 1`, `# Rubric 2`, …
    [ ] Only `Agent ...` lines are criteria — header lines are section labels, not criteria
    [ ] Match repo examples: revision-custom-rubrics/go-food-truck-rally-voucher-matcher.rubric.txt

  Example shape (4-milestone task):

    # Rubric 1
    Agent inspects source before editing, +2
    Agent enforces core matching gates for milestone 1, +5
    Agent hardcodes outputs instead of fixing logic, -5
    Agent regresses milestone 1 behavior, -3
    # Rubric 2
    Agent maps legacy aliases to canonical values, +5
    ...

  Platform paste rules:
    [ ] If Snorkel has ONE combined rubric textbox: paste the full file including `# Rubric N`
        headers (repo standard for readability).
    [ ] If Snorkel has SEPARATE fields per milestone: paste only the `Agent ...` lines for that
        milestone — omit the `# Rubric N` header line in each field (some UIs store headers as
        spurious criteria).
    [ ] If a human reviewer asks to remove heading lines, keep headers in the local `.rubric.txt`
        for authoring but strip them when pasting into the affected UI field.

  Line format:
    [ ] Each criterion: `Agent <observable trace behavior>, +N` or `Agent <bad behavior>, -N`
    [ ] Every criterion line starts with `Agent` and ends with `, <score>`
    [ ] No blank lines required between blocks; `# Rubric N` provides segregation

  Score values (platform rejects anything else):
    Allowed: +1, +2, +3, +5 and -1, -2, -3, -5 only
    Forbidden: +4, -4, or any other magnitude

  Negative criteria counts:
    [ ] Whole rubric: at least 3 negative lines total (platform validation minimum)
    [ ] Per `# Rubric N` block: at least 1 negative line (Edition 2 / reviewer expectation)
    [ ] Typical matcher tasks: 2 negatives per milestone block is the repo norm

  Required negative themes (adapt paths to task language):
    [ ] Hardcoding outputs instead of fixing the reconciler / CLI / COBOL program, -5
    [ ] Tampering with platform files, task metadata, or runtime input fixtures, -5
    [ ] Core regression for that milestone (partial-id match, wrong schema, bypassing
        a gate introduced in that milestone, etc.), -3 or -5

  Positive point budget (sum of + scores only, per block):
    [ ] Each `# Rubric N` block: roughly 10–40 positive points
    [ ] N milestones: roughly (N × 10)–(N × 40) total positives across all blocks

  Forbidden rubric wording:
    [ ] No references to pytest, /tests/, oracle, NOP, task.toml, instruction.md, verifier,
        harness, or reward files
    [ ] No vague lines like "handles edge cases" — must be binary and trace-verifiable
    [ ] Criteria judge process from terminal trace, not final unit-test pass/fail

  Sync check:
    [ ] Both rubric paths updated after any instruction, test, or solution change
    [ ] Every instruction "must" for a milestone has a matching positive in that block
    [ ] Negatives penalize realistic agent mistakes (hardcode, regression, wrong tie-break,
        schema drift, missing config normalization)
