The release packaging pipeline under `/app` is drifting after partial edits. Every verifier scenario writes a packaged drop to `/app/out/pkg/`; that drop must contain only the shared objects named by the active release layout, and `/app/out/pkg/seal.json` must contain an `epoch` string plus a `files` object mapping each shipped filename in `/app/out/pkg/bin/` to its SHA-256 hash. The `epoch` is a 24-character lowercase hex prefix of the build script's provenance fingerprint. The `libplugin_core.so` that ships in the package must be the release-profile build, and the epoch exported from that library must match the manifest epoch.

The shipped `libplugin_core.so` exports three pairs of C symbols that the loader and grader read via dlopen: `plugin_epoch` and `plugin_epoch_len` (string + length, where the string equals the manifest `epoch`); `plugin_build_note` and `plugin_build_note_len` (string + length, where the string starts with `surface-anchor-` and changes whenever the Rust surface or template moves); `plugin_profile_tag` and `plugin_profile_tag_len` (string + length, where the string equals `release-profile` for any shipped build). All six symbols must be preserved across any source-level rewrite.

That epoch is a provenance fingerprint for the shipped release bundle. When a Rust source edit, a codegen-template edit, or a release-layout change alters what should ship, the packaged epoch and the packaged binaries need to move together. Old manifests paired with new binaries, or binaries swapped after packaging, are invalid and the loader at `/app/out/loader` must reject them; on rejection the loader exits non-zero and writes the substring `hash mismatch` to stderr.

Keep incremental rebuilds truthful: do not solve this by forcing unconditional full cleans into every step and do not globally disable incremental Cargo builds. The fix has to hold for the harness scenarios under `/app/ci/harness/`, every sequence in `/app/ci/sequences.json`, and the single-subsystem regression matrix.
