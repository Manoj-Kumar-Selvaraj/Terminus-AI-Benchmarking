The treasury wire adjudicator must now enforce settlement-window and operating-day cutoff controls. Continue to run it through `/app/scripts/run_batch.sh`, and preserve all milestone 1 matching, alias, direction, consumption, ordering, report, and summary behavior.

Use `/app/config/clearing_windows.psv` with pipe-delimited columns `account|open_ts|close_ts|state`. A clearing candidate is time-eligible only when its `account` has a row whose `state` equals runtime `OPEN_CLEAR_STATE`, both timestamps are 14-digit numeric values, `open_ts <= posted_ts <= close_ts`, and `posted_ts <= claim_ts <= close_ts`. The `posted_ts <= claim_ts` requirement is a separate gate from the window bounds; a clearing row whose `posted_ts` is later than the claim's `claim_ts` is ineligible even when both timestamps fall within an open window. Closed, missing, malformed, or wrong-account windows are ineligible.

Also use `/app/config/settlement_calendar.psv` with columns `business_date|cutoff_ts|state`. The business date is the first eight digits of the clearing candidate's `posted_ts` (not `claim_ts`). A candidate clears only when that business date is `OPEN`, both `posted_ts` and `claim_ts` are no later than `cutoff_ts`, and the cutoff itself is a valid 14-digit timestamp. Missing, closed, or malformed calendar rows return the claim without changing the public schemas.

When multiple clearing rows pass all gates for one claim, choose the latest `posted_ts`; if those timestamps are equal, choose the earliest clearing input row. The chosen clearing row is consumed before later claims are adjudicated.
