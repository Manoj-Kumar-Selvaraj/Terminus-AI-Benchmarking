Extend the trip reconciliation CLI in `/app/cmd/reconcile/main.go` to handle dated rebate batches. It must still read `/app/data/trips.csv` and `/app/data/rebates.csv`, then write `/app/out/rebate_report.csv` and `/app/out/rebate_summary.json` with the same schema and status values from the earlier milestones. Input CSV schemas remain the same as milestone 1, with optional `trip_date` appended as the last column on `trips.csv` and `rebate_date` appended as the last column on `rebates.csv` when present.

A rebate can match only when all prior criteria still pass, including full trip_id, rider_id, route_id, amount, canonical mode, and trip status `TAPPED` in `trips.csv` compared case-insensitively (not `POSTED`, `DRAFT`, or other values). The rebate_date must appear in `/app/config/cutoff_calendar.txt` as `YYYY-MM-DD open` (one date per line; `YYYY-MM-DD closed` dates are ineligible), and the rebate_date must not be later than the trip `trip_date`. Missing, unlisted, closed, or blank rebate dates are ineligible; when the rebates CSV includes a `rebate_date` column, an empty value is ineligible (not the same as milestone 1–2 files that omit the column). A trip with a missing or blank `trip_date` is also ineligible. When multiple unused trips match the same rebate, choose the eligible trip with the latest trip_date; ties break to the earliest trip row in file order. Consumption is tracked by trip input row position, not by trip_id.

Legacy aliases from milestone 2 still apply (`BST`→`BUS`, `LRT`→`RAIL`, `FRY`→`FERRY`); matched report rows must emit the canonical mode and unmatched rows must leave `mode` blank. Summary amounts remain positive integer cents.
