Update the trip reconciliation CLI in `/app` for a legacy rebate export that uses mode aliases. The program must still read `/app/data/trips.csv` and `/app/data/rebates.csv`, then write `/app/out/rebate_report.csv` and `/app/out/rebate_summary.json`. Input CSV schemas remain the same as milestone 1: `trips.csv` uses `trip_id,rider_id,route_id,amount_cents,status,mode` and `rebates.csv` uses `trip_id,rider_id,route_id,amount_cents,mode`.

A rebate matches only when trip_id, rider_id, route_id, amount, trip status, and allowed mode line up. In `trips.csv`, the only eligible trip status is `TAPPED` compared case-insensitively; do not treat `POSTED`, `DRAFT`, or any other status as eligible. Trip_ids must be compared as full identifiers, fields may have surrounding spaces, and mode and status comparisons should be case-insensitive. Modes `BUS`, `RAIL`, and `FERRY` are allowed, and legacy rebate mode aliases `BST`, `LRT`, and `FRY` should normalize to canonical `BUS`, `RAIL`, and `FERRY`.

Each trip can be consumed by at most one rebate, with the earliest eligible rebate winning. The report must keep rebate input order with columns `trip_id,rider_id,mode,amount_cents,status`; matched rows should emit the canonical mode, while unmatched rows must leave `mode` blank. The summary JSON must contain integer fields `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents`, with rebate amounts counted as positive cents.
