Extend the device warranty claim reconciler in `/app/cmd/reconcile/main.go` to handle dated warranty claim batches. It must still read `/app/data/devices.csv` and `/app/data/warranty_claims.csv`, then write `/app/out/warranty_report.csv` and `/app/out/warranty_summary.json` with the same report schema and summary schema from the earlier milestones. Report statuses are exactly `MATCHED` and `UNMATCHED`. The input files may add a `purchase_date` column to `devices.csv` and a `claim_date` column to `warranty_claims.csv`. Older CSV shapes without those columns must still be readable without crashing, but they are readable only for reporting: a device row without `purchase_date`, or a claim row without `claim_date`, is not eligible to match and must produce `UNMATCHED` claim rows rather than falling back to milestone 1 or milestone 2 matching. A claim can match only when all prior criteria still pass, the claim date is listed as `open` in `/app/config/cutoff_calendar.txt`, and the device purchase date is not later than the claim date. A missing or closed claim date is not eligible. If more than one unused device row matches the same claim, choose the eligible row with the latest purchase date; if purchase dates tie, choose the earliest device row. Consumption is tracked per input row, not per `device_id`: duplicate device rows with the same `device_id` are independently consumable, but the same physical CSV row can still be consumed at most once. Legacy aliases from milestone 2 still apply (`BAT` means `BATTERY`, `WTR` means `WATER`), and matched report rows must emit the canonical reason. Unmatched rows must leave `reason` blank. Summary amounts remain positive integer cents.
