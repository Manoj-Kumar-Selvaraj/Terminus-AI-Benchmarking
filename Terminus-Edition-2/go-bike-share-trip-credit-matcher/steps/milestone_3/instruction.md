Extend the trip reconciliation CLI in `/app/cmd/reconcile/main.go` to handle dated credit batches. It must still read `/app/data/trips.csv` and `/app/data/credits.csv`, then write `/app/out/credit_report.csv` and `/app/out/credit_summary.json` with the same schema and status values from the earlier milestones. dated inputs use the schemas `trips.csv`: `trip_id,rider_id,amount_cents,status,pass_type,ride_date` and `credits.csv`: `trip_id,rider_id,amount_cents,pass_type,credit_date` (no `station_id` column when date columns are present). All date fields and calendar rows use `YYYY-MM-DD` format. A credit can match only when all prior criteria still pass except station matching, which is not required in dated mode, the credit_date is listed as `open` in `/app/config/cutoff_calendar.txt`, and the credit_date is not later than the trip ride_date. A missing or closed credit_date is not eligible. A trip with a missing ride_date is also not eligible because the date comparison cannot be satisfied. If more than one unused trip matches the same credit, choose the eligible trip with the latest ride_date; if ride_dates tie, choose the earliest trip row. Consumption is tracked by trip input row position, not by trip_id, so duplicate ids in separate rows remain independently consumable until their specific rows are used. Legacy aliases from milestone 2 still apply (`DY` means `DAY`, `MO` means `MONTH`, `AN` means `ANNUAL`), and matched report rows must emit the canonical pass_type. Unmatched rows must leave `pass_type` blank. Summary amounts remain positive integer cents.
