Repair the childcare refund reconciliation CLI in `/app`. The executable is built from `/app/cmd/reconcile` with the Go toolchain available at `/usr/local/go/bin/go`. It must read `/app/data/sessions.csv` and `/app/data/refunds.csv`, regenerate `/app/out/refund_report.csv`, and regenerate `/app/out/refund_summary.json` on every run.

For this milestone, CSV files are header-addressed: columns may appear in any order and may include extra columns. A session row needs `session_id`, `guardian_id`, `amount_cents`, `status`, and `room`; a refund row needs `session_id`, `guardian_id`, `amount_cents`, and `room`. Trim incidental surrounding whitespace from input fields. Compare `status` and `room` case-insensitively. Treat only positive integer cent amounts as eligible for matching. Session rows with missing, malformed, or non-positive required fields are ignored as candidates. Refund rows with missing, malformed, or non-positive amounts must still appear in the report as `UNMATCHED`; they contribute `0` cents to summary totals. Write the trimmed but otherwise unparsed `amount_cents` text in the report column, preserving malformed or empty values after trimming only. If a required CSV column is absent from a file header, treat every row from that file as if the field were empty rather than aborting.

A refund matches a session only when full `session_id`, `guardian_id`, positive `amount_cents`, `CHECKEDIN` session status, and an allowed room all line up. Allowed rooms are exactly `INFANT`, `TODDLER`, and `PREK` after trim and case folding. Do not use prefix, substring, or fuzzy identifier matching. Each session input row can be consumed at most once; when more than one refund could consume the same row, refund input order wins.

The report must keep one row per refund in refund input order with columns exactly `session_id,guardian_id,room,amount_cents,status`. Use the cleaned refund `session_id` and `guardian_id`, emit the canonical room for matched rows, leave `room` blank for unmatched rows, and use only `MATCHED` or `UNMATCHED` in `status`. The summary JSON must contain integer fields `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents`, with refund amounts counted as positive cents. Do not add extra report columns or summary keys.
