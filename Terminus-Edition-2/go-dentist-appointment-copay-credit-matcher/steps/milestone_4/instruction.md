Extend the appointment credit reconciler in `/app/cmd/reconcile/main.go` with configured type policy from `/app/config/methods.csv`, patient daily credit limits from `/app/config/patient_limits.csv`, and blackout-date exclusions from `/app/config/blackouts.csv` while preserving every milestone 1, milestone 2, and milestone 3 rule and output schema.

The methods file is a CSV with columns `service_type`, `enabled`, and optional `priority`. Normalize `service_type` after trimming and case folding using the same alias rules as credits (`CL` to `CLEAN`, `XR` to `XRAY`, `SG` to `SURG`). A selected canonical type is eligible only when `methods.csv` contains a row whose normalized type is canonical and whose `enabled` value is `true` case-insensitively. Lower `priority` numbers rank earlier; missing or malformed priorities rank after configured numeric priorities.

Credits may use `ANY` as the `service_type`. `ANY` is never emitted in the report. An `ANY` credit can match any enabled appointment service type that satisfies the appointment id, patient id, amount, active status, date, and consumption rules. For `ANY`, select the unused eligible appointment with the latest `appointment_date` when date columns are present; if class dates tie, select the lower configured class-type priority; if priority also ties, select the earliest appointment input row. When date columns are absent, rank only by priority then earliest appointment row. Non-`ANY` credits still require the canonical service type to match exactly and must be enabled in the methods file.

`patient_limits.csv` is a CSV with columns `patient_id,service_type,effective_date,max_daily_amount,status`. Normalize `service_type` after trimming and case folding with the same aliases as credits and methods. A dated action can match only if all prior gates pass and an `ACTIVE` limit row matches the action identity field and selected canonical type, has numeric `max_daily_amount`, and has `effective_date` on or before the action date. Rows that do not satisfy that definition do not apply. If multiple limit rows match, choose the one with the latest `effective_date`; if dates tie, choose the earliest limit input row.

The chosen limit is a daily cap for matched credits with the same `patient_id`, selected canonical service type, and `credit_date`. Matched credits consume that cap in credit input order. A credit that would make the consumed total exceed `max_daily_amount` must remain `UNMATCHED` and must not consume a appointment row or budget. `ANY` credits use the service type of the selected candidate appointment for limit lookup and budget consumption. Apply the budget gate only in dated mode when `credit_date` exists; undated inputs continue to follow methods and `ANY` behavior without patient-limit checks.

`blackouts.csv` is a CSV with columns `service_type,start_date,end_date,state`. Normalize `service_type` after trimming and case folding with the same aliases as credits, methods, and limits. A dated action cannot match a source when the selected canonical type has an `ACTIVE` blackout row with valid `YYYY-MM-DD` `start_date` and `end_date` and `start_date <= action date <= end_date`. Rows that do not satisfy that definition do not apply. If any valid active blackout applies, that candidate appointment is ineligible.

For `ANY` credits, blackout filtering happens before candidate ranking by latest `appointment_date`, configured priority, and earliest appointment row. If the highest-priority or latest-date candidate is blacked out, choose the next eligible candidate that still satisfies every prior gate, method policy, calendar rule, and patient limit. A credit rejected only because all otherwise valid candidates are blacked out remains `UNMATCHED` and consumes no appointment row or budget.

When date columns are absent, skip blackout and patient-limit gates and keep methods and `ANY` undated behavior.
