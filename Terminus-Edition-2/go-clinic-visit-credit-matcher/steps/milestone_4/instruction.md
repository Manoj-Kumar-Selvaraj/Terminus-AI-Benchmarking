Extend the visit credit reconciler in `/app` so channel eligibility is controlled by the runtime method configuration. The program must still read `/app/data/visits.csv` and `/app/data/credits.csv`, then write `/app/out/credit_report.csv` and `/app/out/credit_summary.json` with the same report columns, JSON fields, status labels, row ordering, positive-cent accounting, alias behavior, date behavior, and one-row consumption rules from the earlier milestones.

In addition to the prior matching gates, a candidate visit channel is eligible only when its canonical channel is enabled in `/app/config/methods.csv`. The method file uses the schema `channel,enabled`. Trim surrounding whitespace from both fields and compare them case-insensitively. A malformed method row has any field count other than exactly two columns. Only an enabled value of `true` makes the canonical channel eligible; missing rows, disabled rows, malformed rows, blank channel names, unsupported channel names, and any non-true enabled value are ineligible. Method rows are keyed by canonical channel names only — do not apply credit-file alias normalization to `methods.csv` keys.

Apply method eligibility after channel alias normalization, so legacy credit aliases `CC` and `WIR` are evaluated as canonical `CARD` and `WIRE`. Method rows are keyed by canonical channel names, and matched report rows must continue to emit canonical `ACH`, `CARD`, or `WIRE` values. The methods gate applies in both legacy undated mode and dated mode. Dated mode still uses `/app/config/cutoff_calendar.txt`: when date columns are present, the credit date must be listed as open and not later than the candidate visit due date; when neither date column is present, skip date gating but still apply method eligibility.
