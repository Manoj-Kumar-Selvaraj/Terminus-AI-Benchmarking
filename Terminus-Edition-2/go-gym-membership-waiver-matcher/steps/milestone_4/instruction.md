Extend the membership reconciliation CLI in `/app/cmd/reconcile/main.go` for waiver method policy gating while preserving all earlier matching behavior and output schemas. The program still reads `/app/data/memberships.csv` and `/app/data/waivers.csv`, then writes `/app/out/waiver_report.csv` and `/app/out/waiver_summary.json`.

The input files may add a `waiver_method` column to `waivers.csv`. When that column is present, a waiver can match only if its method is enabled in `/app/config/methods.csv`. The methods config uses CSV columns `method,enabled`. Compare method names after trimming and case folding. A method is enabled only when the config row's `enabled` value is `true`, also after trimming and case folding. Disabled methods, methods missing from the config, blank waiver methods, blank config methods, malformed config rows, or non-true enabled values such as `yes`, `1`, or `enabled` make the waiver ineligible. If `waiver_method` is absent from the waiver input header, keep milestone 3 behavior without method gating.

Method gating must compose with all prior gates. An enabled method alone must not allow a waiver to match when identity, amount, status, plan, date, calendar, or consumption rules fail. The report schema remains `membership_id,member_id,plan,amount_cents,status`; matched rows still emit the canonical plan, and unmatched rows still leave `plan` blank.
