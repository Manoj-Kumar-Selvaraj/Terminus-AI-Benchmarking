The membership reconciliation CLI in `/app/cmd/reconcile/main.go` is misclassifying a few same-day waivers. PLUS waivers are being left unmatched, one membership id collision is matching the wrong row, and the matched total is coming out with the wrong sign. Fix the Go program so it reads `/app/data/memberships.csv` and `/app/data/waivers.csv`, then writes `/app/out/waiver_report.csv` and `/app/out/waiver_summary.json`. A waiver matches only when membership_id, member_id, amount, posted membership status, and plan all line up. The membership plan and waiver plan must equal each other after trimming and case normalization, and both must resolve to an allowed value. Plans `BASIC`, `PLUS`, and `ELITE` are allowed. Each membership can be matched at most once; duplicate waivers targeting the same membership leave only the earliest eligible waiver matched. Input fields can have incidental surrounding spaces, and plan and status comparisons should be case-insensitive. The report must keep one row per waiver in input order with columns `membership_id,member_id,plan,amount_cents,status`; use `MATCHED` or `UNMATCHED` for status, use the waiver's plan when matched, and leave `plan` blank when no membership matched. Report fields should not carry incidental surrounding spaces from the input. The summary JSON must contain integer fields `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents`, with waiver amounts counted as positive cents.
