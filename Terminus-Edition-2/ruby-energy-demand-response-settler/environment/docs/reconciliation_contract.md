# Demand Response Settlement Contract

The executable entrypoint is `/app/app/reconcile.rb`; verifiers run `ruby /app/app/reconcile.rb`. The program reads `/app/data/settlements.csv` as correction rows and `/app/data/events.csv` as source rows. Output files are `/app/out/cod_demand_response_report.csv` and `/app/out/cod_demand_response_summary.txt`. Milestones 3 and 4 also write `/app/out/event_consumption.csv`.

Core matching requires full equality for `parcel_id`, `meter_id`, `station_id`, `feeder`, and `amount`; source status `CONFIRMED`; correction reason `CURTAIL`, `BONUS`, or `CORRECT`; numeric timestamps; `settle_ts` on or after `event_ts`; and one-time source-row consumption. Candidate ordering is latest eligible `event_ts`, then earliest physical source input row. Reports preserve settlement input order, use only `MATCHED` or `UNMATCHED`, emit the selected canonical source `resource_type` for matched rows, leave unmatched `resource_type` blank, and count summary amounts as positive integers.

Milestone 1 has no legacy aliases and only canonical `LOAD` or `SOLAR` resource types are match-eligible. `/app/config/windows.csv` is not part of milestone 1 eligibility.

Milestone 2 adds `BATTERY` and legacy aliases: `LD` means `LOAD`, `QR` means `SOLAR`, and `CC` means `BATTERY`. Alias normalization trims and case-folds both source and settlement `resource_type` values before validation, matching, policy lookup, and report output. Unknown normalized resource types stay unmatched.

Milestone 3 makes `/app/config/windows.csv` authoritative. A source timestamp must be inside an explicitly `OPEN` window for the same `station_id`, and the settlement timestamp must be on or after the source timestamp and not after the window close. Closed, missing, malformed, or unlisted windows are ineligible. `/app/out/event_consumption.csv` has header `settlement_id,event_row` and records one zero-based physical source data-row position for each matched settlement in settlement input order.

Milestone 4 adds `/app/config/resource_policy.csv` with columns `station_id,resource_type,enabled,priority,max_station_amount`. Source candidates require a policy row for the same station and canonical source resource type, `enabled` equal to `true` case-insensitively, numeric `priority`, and numeric `max_station_amount`. Settlement `resource_type` `ANY` may match any enabled canonical source type after all other gates. Named resource types require exact canonical equality. For `ANY`, choose latest source timestamp first, then lowest numeric policy priority, then earliest source row. Apply `max_station_amount` as a running cap per `(station_id, resource_type)` in settlement input order.

Milestone 5 adds `/app/config/settlement_overrides.csv` with columns `settlement_id,mode,resource_type,expires_ts`. Override rows are valid only when `settlement_id` matches the settlement, `expires_ts` is numeric, and `expires_ts` is greater than or equal to the settlement `settle_ts`. Valid `DENY` rows make that settlement unmatched before candidate selection and do not consume a source row. Valid `FORCE_RESOURCE` rows replace the settlement's effective resource type with a canonical `LOAD`, `SOLAR`, or `BATTERY` value before candidate selection; invalid forced resource types are ignored. When multiple valid overrides exist for a settlement, any valid `DENY` wins; otherwise use the valid `FORCE_RESOURCE` row with the latest `expires_ts`, breaking equal expiry ties by earliest override input row. Overrides never bypass identity, timestamp, window, policy, budget, or consumption gates.
