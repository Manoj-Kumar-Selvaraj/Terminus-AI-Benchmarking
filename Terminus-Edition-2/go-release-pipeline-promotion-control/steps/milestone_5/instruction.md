Rollback now requires a signed immutable release-history envelope. Preserve all milestone 1-4 behavior and the existing command interface `go run ./cmd/pipelinesim rollback --history PATH --env ENV --out DIR [--target-build BUILD]`, but before selecting any default or explicit rollback target, verify a sibling signature file at `PATH + ".sig"` using `/app/config/release_history_keys.json`.

The key config has schema `release-history-keys/v1` and a `keys_by_env` object. Each environment entry has `key_id` and `secret`. The signature file has schema `release-history-signature/v1`, `algorithm` exactly `HMAC-SHA256`, `environment`, `key_id`, `history_digest`, and `signature`. The signature environment must equal the requested rollback `--env`, the key id must equal the configured key id for that environment, and `history_digest` must equal the SHA-256 hex digest of the same canonical release-history part stream used for the HMAC. Missing key config, missing signature file, malformed JSON, unsupported schema, unsupported algorithm, unknown environment, wrong key id, missing or stale history digest, or signature mismatch must fail closed without writing `rollback_manifest.json`. A failed signed rollback must also remove any stale `rollback_manifest.json` already present in the requested output directory before returning failure.

Compute the signature over the full parsed release history in file order, not just the finally selected record. Use HMAC-SHA256 with the configured secret. Feed length-delimited fields exactly like the simulator digest contract: for each string part, write the UTF-8 bytes followed by a single zero byte. The canonical part sequence is `release-history/v1`, the history `schema_version`, then for every release record in order: `environment`, `build_number`, `commit_sha`, `artifact_hash`, `promoted_artifact_hash`, `package_hash`, `promotion_status`, `release_contract_version`, and the decimal count of `deployment_units` (use `0` when absent). For each deployment unit in the original array order, append `name`, `artifact_hash`, `package_hash`, and `release_contract_version`, with missing fields represented as empty strings. Compute `history_digest` as plain SHA-256 over that same zero-delimited part stream, and compute `signature` as HMAC-SHA256 over that stream with the configured secret. Compare both expected hex values to the submitted hex values case-insensitively.

After signature verification succeeds, apply the milestone 4 rollback rules unchanged and extend the compatibility floor to multi-artifact bundle records. A release record that includes `deployment_units` is eligible only when every unit has non-empty `name`, `artifact_hash`, `package_hash`, and a well-formed `release_contract_version` greater than or equal to the requested environment's floor. Successful rollback must copy the selected history record's `deployment_units` array verbatim, preserving order, into `rollback_manifest.json` in addition to copying `artifact_hash` and `promoted_artifact_hash` independently from the selected history literal.
