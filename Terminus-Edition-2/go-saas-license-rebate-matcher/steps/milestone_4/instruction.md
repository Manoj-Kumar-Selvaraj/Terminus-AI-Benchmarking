Extend the SaaS license rebate reconciler with configured tier policy controls. The program must still run as the Go CLI in `/app/cmd/reconcile/main.go`, read `/app/data/licenses.csv` and `/app/data/rebates.csv`, and write `/app/out/rebate_report.csv` and `/app/out/rebate_summary.json` with the same report schema, summary schema, and `MATCHED` / `UNMATCHED` status values from the earlier milestones.

All milestone 1 through 3 rules still apply: full identifier matching, tenant and amount gates, source status `LICENSED`, canonical tier aliases (`STR`, `BUS`, `ENT`), single-use license row consumption, open calendar dates for dated batches, `rebate_date <= license_end`, and latest `license_end` candidate selection. In this milestone, `/app/config/tier_policy.csv` defines canonical tiers, an enabled flag, and a numeric priority. Treat enabled value `true` as enabled and `false` as disabled. A license tier is eligible only when its canonical tier is enabled in that policy. Rebate tier `ANY` is a wildcard that can match any enabled canonical license tier, but matched report rows must emit the selected canonical tier, never `ANY`. When an `ANY` rebate has multiple eligible licenses, choose by latest `license_end` first; only if the license_end values tie, choose the lower configured priority number, then earliest license input row.

For dated batches, also enforce `/app/config/tenant_limits.csv` and `/app/config/license_blackouts.csv`. Tenant limits apply by `tenant_id` and `rebate_date`; use the latest effective limit whose `effective_date` is not after the rebate date, and if two rows have the same effective date, the later row in the file wins. A dated rebate with no applicable tenant limit row is not eligible to match. Process rebates in input order, and do not match a rebate when adding its amount would exceed that tenant/date budget. A rebate rejected by a limit must not consume a license row. Blackouts apply before candidate ranking: if a license row's `license_id` is in a blackout row and its `license_end` falls inside the inclusive blackout window, that license row is ineligible. Ignore blackout rows with a blank start date, a blank end date, or a start date after the end date.

The verifier may also provide older undated CSVs without `license_end` and `rebate_date`. Those undated inputs should continue to run without crashing, keep the milestone 2 matching rules, and still apply enabled tier policy and `ANY` wildcard selection. Tenant limits and license blackouts apply only when the dated columns are present.
