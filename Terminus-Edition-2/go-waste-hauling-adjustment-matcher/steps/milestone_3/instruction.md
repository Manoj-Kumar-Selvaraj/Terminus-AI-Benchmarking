Extend the haul reconciliation CLI in `/app/cmd/reconcile/main.go` to handle dated adjustment batches. It must still read `/app/data/hauls.csv` and `/app/data/adjustments.csv`, then write `/app/out/adjustment_report.csv` and `/app/out/adjustment_summary.json` with the same schema and status values as before. Build the deliverable as a Go CLI with the toolchain at `/usr/local/go/bin/go`.

Date rules apply only when both files include their date column (`service_date` in hauls and `adjustment_date` in adjustments). If either column is absent, ignore `/app/config/cutoff_calendar.txt` and match exactly as before (aliases, consumption, and all prior gates still apply). When both date columns are present, an adjustment can match only when all prior criteria still pass, the `adjustment_date` is listed as `open` in `/app/config/cutoff_calendar.txt`, and the `adjustment_date` is not later than the haul `service_date`. Missing, closed, or unlisted `adjustment_date` values are not eligible. A haul with missing `service_date` is not eligible in dated mode. If more than one unused haul matches the same adjustment, choose the eligible haul with the latest `service_date`; if `service_date` values tie, choose the earliest haul input row. Consumption is tracked by haul input row position, not by `haul_id`, so duplicate ids in separate rows remain independently consumable until their specific rows are used.

Legacy aliases still apply (`RES` means `RESI`, `COM` means `COMM`, `INDL` means `IND`), and matched report rows must emit the canonical route. Unmatched rows must leave `route` blank. Summary amounts remain positive integer cents.
