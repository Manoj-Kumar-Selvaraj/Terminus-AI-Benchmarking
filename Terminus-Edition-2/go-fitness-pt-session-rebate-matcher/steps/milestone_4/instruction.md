Extend the session rebate reconciler in `/app/cmd/reconcile/main.go` with configured type policy from `/app/config/methods.csv`, client daily rebate limits from `/app/config/client_limits.csv`, and blackout-date exclusions from `/app/config/blackouts.csv` while preserving every milestone 1, milestone 2, and milestone 3 rule and output schema.

The methods file is a CSV with columns `training_type`, `enabled`, and optional `priority`. Normalize `training_type` after trimming and case folding using the same alias rules as rebates (`SO` to `SOLO`, `DU` to `DUO`, `TM` to `TEAM`). A selected canonical type is eligible only when `methods.csv` contains a row whose normalized type is canonical and whose `enabled` value is `true` case-insensitively. Lower `priority` numbers rank earlier; missing or malformed priorities rank after configured numeric priorities.

Rebates may use `ANY` as the `training_type`. `ANY` is never emitted in the report. An `ANY` rebate can match any enabled session training type that satisfies the session id, client id, amount, active status, date, and consumption rules. For `ANY`, select the unused eligible session with the latest `session_date` when date columns are present; if class dates tie, select the lower configured class-type priority; if priority also ties, select the earliest session input row. When date columns are absent, rank only by priority then earliest session row. Non-`ANY` rebates still require the canonical training type to match exactly and must be enabled in the methods file.

`client_limits.csv` is a CSV with columns `client_id,training_type,effective_date,max_daily_amount,status`. Normalize `training_type` after trimming and case folding with the same aliases as rebates and methods. A dated action can match only if all prior gates pass and an `ACTIVE` limit row matches the action identity field and selected canonical type, has numeric `max_daily_amount`, and has `effective_date` on or before the action date. Rows that do not satisfy that definition do not apply. If multiple limit rows match, choose the one with the latest `effective_date`; if dates tie, choose the earliest limit input row.

The chosen limit is a daily cap for matched rebates with the same `client_id`, selected canonical training type, and `rebate_date`. Matched rebates consume that cap in rebate input order. A rebate that would make the consumed total exceed `max_daily_amount` must remain `UNMATCHED` and must not consume a session row or budget. `ANY` rebates use the training type of the selected candidate session for limit lookup and budget consumption. Apply the budget gate only in dated mode when `rebate_date` exists; undated inputs continue to follow methods and `ANY` behavior without client-limit checks.

`blackouts.csv` is a CSV with columns `training_type,start_date,end_date,state`. Normalize `training_type` after trimming and case folding with the same aliases as rebates, methods, and limits. A dated action cannot match a source when the selected canonical type has an `ACTIVE` blackout row with valid `YYYY-MM-DD` `start_date` and `end_date` and `start_date <= action date <= end_date`. Rows that do not satisfy that definition do not apply. If any valid active blackout applies, that candidate session is ineligible.

For `ANY` rebates, blackout filtering happens before candidate ranking by latest `session_date`, configured priority, and earliest session row. If the highest-priority or latest-date candidate is blacked out, choose the next eligible candidate that still satisfies every prior gate, method policy, calendar rule, and client limit. A rebate rejected only because all otherwise valid candidates are blacked out remains `UNMATCHED` and consumes no session row or budget.

When date columns are absent, skip blackout and client-limit gates and keep methods and `ANY` undated behavior.
