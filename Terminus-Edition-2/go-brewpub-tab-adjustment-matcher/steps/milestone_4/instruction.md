Extend the tab reconciliation CLI in `/app/cmd/reconcile/main.go` to handle method-gated adjustment batches. It must still read `/app/data/tabs.csv` and `/app/data/adjustments.csv`, then write `/app/out/tab_adjustment_report.csv` and `/app/out/tab_adjustment_summary.json` with the same report and summary schemas from the earlier milestones. Keep the deliverable as a Go CLI: the verifier compiles the Go source with the Go toolchain available at `/usr/local/go/bin/go` and then runs the produced binary.

All milestone 1 through 3 behavior remains required: tab ids must match as full identifiers; patron_id, amount, posted tab status, allowed pour_tier, legacy pour_tier aliases, open adjust_date, adjust_date not later than tab_date, latest eligible tab_date selection, same-date input-order tie-breaking, and per-tab-row consumption all still apply. Posted tab status is stored as `COMPLETED` in `tabs.csv`. Legacy credit pour_tier aliases still normalize case-insensitively as `PT` to `PINT`, `PC` to `PITCH`, and `KG` to `KEG`, and matched rows must emit the canonical credit pour_tier.

For this milestone, `adjustments.csv` may include an `adjust_method` column after `adjust_date`. When that column is present, a credit is eligible only if its trimmed, case-folded adjustment method appears in `/app/config/methods.csv` with an `enabled` value that trims and case-folds exactly to `true`. The methods file has columns `method,enabled`. Disabled methods, missing methods, blank adjustment methods, blank method rows, and malformed rows whose enabled value is anything other than `true` (such as `yes` or `false`) are not eligible to match. The verifier supplies `methods.csv` rows with surrounding spaces on method names and enabled tokens that must still trim and case-fold before comparison. Method gating is additive: an enabled method never bypasses status, amount, pour tier, calendar, tie-break, or row-consumption requirements. Method names should be compared case-insensitively after trimming. If the `adjust_method` column is absent, preserve milestone 3 behavior without applying the method gate.

The output schema does not change for method-gated batches. The verifier expects `tab_adjustment_report.csv` to begin with exactly `tab_id,patron_id,pour_tier,amount_cents,status` with no `adjust_method` column, in adjustment input order. Unmatched rows must leave `pour_tier` blank, and summary amounts remain positive integer cents. Adjustment methods listed only in `adjustments.csv` but absent from `methods.csv` are not eligible.
