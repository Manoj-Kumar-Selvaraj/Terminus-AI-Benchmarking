The tab reconciliation CLI in `/app/cmd/reconcile/main.go` is misclassifying a few same-day tab adjustments. The starter still uses legacy helper and type names (`loadTripes`, `Trip`, `Credit`, `PassType`) even though the CSV columns are `tab_id`, `patron_id`, and `pour_tier`. PITCH adjustments are being left unmatched, one tab id collision is matching the wrong row, the matched total is coming out with the wrong sign, and the summary is written to the wrong output path. Keep the deliverable as a Go CLI: the verifier compiles the Go source with the Go toolchain available at `/usr/local/go/bin/go` and then runs the produced binary.

Fix the Go program so it reads `/app/data/tabs.csv` and `/app/data/adjustments.csv`, then writes `/app/out/tab_adjustment_report.csv` and `/app/out/tab_adjustment_summary.json` (not `/app/out/credit_summary.json`). A tab adjustment matches only when `tab_id`, `patron_id`, `amount_cents`, eligible tab status `COMPLETED` (compared case-insensitively), and pour tier all line up. Pour tiers `PINT`, `PITCH`, and `KEG` are allowed on both sides of the match. Each tab row can be consumed at most once; if duplicate adjustments target the same tab row, only the earliest eligible adjustment in input order consumes it. Input fields may have incidental surrounding spaces, and pour tier and status comparisons should be case-insensitive.

The report must keep one row per adjustment in input order. The verifier expects `tab_adjustment_report.csv` to begin with the header line exactly `tab_id,patron_id,pour_tier,amount_cents,status`, uses `MATCHED` or `UNMATCHED` for status, and leaves `pour_tier` blank on unmatched rows. Matched rows must emit the canonical adjustment pour tier from the adjustment row (trimmed and uppercased), not the tab row's stored pour tier when they differ. Report `tab_id`, `patron_id`, and `amount_cents` must not carry incidental surrounding spaces from adjustment input. The summary JSON at `/app/out/tab_adjustment_summary.json` must contain only integer-typed fields `matched_count`, `matched_amount_cents`, `unmatched_count`, and `unmatched_amount_cents`, with positive cent totals.

Header-only input files are valid: an empty `adjustments.csv` yields only the report header and zero summary totals, while adjustments against a `tabs.csv` file with only a header remain unmatched with positive unmatched totals.
